# Agents Guide

## Project Overview

This repository is a Python 3.12 application (pinned to `3.12.8` via `.python-version`) that turns an Ableton Push 2 into a controller for Logic Pro. It extends the original Psyha-style standalone Push 2 controller with Logic-specific transport, keyboard shortcut, Mackie Control/MCU, MIDI CC, scale, preset, and display workflows.

The runtime entry point is `app.py` (~62 KB / ~1300 lines). It initializes MIDI ports, the Push 2 device via `push2_python`, the Logic/MCU integration, all controller modes, and then runs a continuous display/event loop. All module-level `@push2_python.*` global callback bindings live at the bottom of `app.py`.

This project is hardware- and DAW-facing. Many changes cannot be fully validated without:

- macOS with Logic Pro configured.
- Ableton Push 2 connected.
- IAC Driver MIDI ports configured as described in `README.md`.
- The Mackie Control surface in Logic routed to the expected IAC ports.

## How To Run

Use the repository virtualenv when present:

```bash
./run.sh
```

Equivalent manual run:

```bash
source .venv/bin/activate
python app.py
```

Install dependencies from the lock file:

```bash
python -m pip install -r requirements.txt
```

`requirements.in` is the top-level dependency input. `requirements.txt` is generated by `pip-compile` (from `pip-tools`). To update locked deps, edit `requirements.in` then re-run `pip-compile`.

### Dependency notes

- **`push2-python` is pinned to a custom fork**: `git+https://github.com/zurie/push2-python@<commit>`. Several constants used in this repo (e.g. `BUTTON_DOUBLE_LOOP`, `BUTTON_ROW_n_COL_n`) only exist in this fork â€” installing upstream `push2-python` from PyPI will break the app.
- `flask`, `flask-socketio`, `eventlet`, `oscpy` are listed as top-level deps but no web UI / OSC bridge is wired into `app.py` in the current tree. Treat them as vestigial / for a future split-view UI (the `requirements.in` header references "SplitView") unless you find live usages.
- `pynput` is used by `logic_keystrokes.py` to send macOS keystrokes â€” this requires macOS Accessibility permission for the terminal/IDE running the app.

## Important Runtime Files

- `settings.json`: persisted user/runtime settings. Notable keys include `use_mcu` (toggles `LogicMCUManager` vs `LogicMidiListener`), `mcu_port_name`, `default_midi_out_device_name`, `gui_profile` (currently `legacy` or `mackie_v2`), `target_frame_rate`, `use_push2_display`, `debug_mcu`, `debug_logs`, `collapse_scale`, `use_poly_at`, `root_midi_note`, and aftertouch curve parameters.
- `push2_keymap.json`: maps `/push2/...` command paths to macOS keystrokes used by `logic_keystrokes.py`.
- `track_listing.json`: ordered list of track/instrument short names used to build Push track selectors.
- `button_docs.json`: per-button help text consumed by `help_mode.py`. Path constant: `HELP_DOCS_PATH` in `definitions.py`.
- `instrument_definitions/*.json`: instrument metadata and MIDI CC mappings used by track selection and MIDI CC mode.
- `device_definitions/*.json`: additional synth/device MIDI CC definition files.
- `favourite_presets.json`: persisted preset favorites for preset selection workflows.

Treat these JSON files as user-facing configuration, not generated throwaway state.

## Architecture

### `app.py`

`LogicApp` is the central coordinator. It:

- Loads settings and configures MIDI input/output.
- Creates `push2_python.Push2`.
- Starts either `LogicMCUManager` or the simpler `LogicMidiListener`.
- Initializes all controller modes.
- Dispatches Push 2 events to active modes in reverse active order.
- Manages mutually exclusive mode groups through `xor_group`.
- Draws the Push 2 display using Cairo.
- Persists settings with `save_current_settings_to_file()`.

Global Push callbacks are registered at module level with `push2_python` decorators. These callbacks implement raw press/release handling plus processed single, double, and long press detection before forwarding to modes.

> Historical note: `self.rhyhtmic_mode` (misspelled) was renamed to `self.rhythmic_mode` in May 2026. All references in `app.py` and `track_selection_mode.py` were updated in the same sweep.

### `definitions.py`

This is more than a base-class module. It is the central kitchen-sink for:

- `LogicMode` base class (all modes inherit from this).
- MCU SysEx prefixes and IDs: `MCU_MODEL_ID = 0x14`, `ACCEPTED_MCU_MODEL_IDS = (0x12, 0x14)`, `MCU_SYSEX_PREFIX`, `MCU_DEVICE_INQUIRY`, `MCU_METERS_ON/OFF`, `MCU_FLIP`.
- MCU modifier note numbers: `MCU_SHIFT`, `MCU_CTRL`, `MCU_OPTION`, `MCU_ALT`.
- Push 2 color palette (`*_RGB` lists and `COLORS_NAMES` with auto-generated `_darker1`/`_darker2` variants), `MIXER_PALETTE`.
- `SCALES` tuple (Major, Minor, Dorian, â€¦ 25 scales) used by melodic mode and scale menu.
- `_FADER_TABLE` plus `pb_to_db()` / `db_to_pb()` â€” 14-bit pitch-bend â†” dB interpolation used for MCU fader work.
- Timing constants: `BUTTON_LONG_PRESS_TIME = 0.25`, `BUTTON_DOUBLE_PRESS_TIME = 0.2`, `DELAYED_ACTIONS_APPLY_TIME = 1.0`, `NOTIFICATION_TIME = 3`, `HELP_TIME = 30`.
- Path constants: `INSTRUMENT_DEFINITION_FOLDER`, `DEVICE_DEFINITION_FOLDER`, `TRACK_LISTING_PATH`, `HELP_DOCS_PATH`.
- Debug HUD toggles for the Mackie display: `MC_DRAW_DEBUG`, `MC_DEBUG_LAYOUT`, `MC_DEBUG_GUIDES`, `MC_DEBUG_FONT`, `MC_DEBUG_HEIGHT`, `MC_SCRIBBLE_TEST`, etc.
- `Timer` / `delay` decorator for delayed actions.

### Mode System

All modes inherit from `definitions.LogicMode`. Important hooks include:

- `initialize(settings=None)`
- `activate()` / `deactivate()`
- `update_pads()`
- `update_buttons()`
- `update_display(ctx, w, h)`
- `on_button_pressed(...)`
- `on_pad_pressed(...)`
- `on_encoder_rotated(...)`
- `on_midi_in(...)`

Modes should return `True` from input handlers when they handled an event and want propagation to stop.

The `xor_group` field controls mutually exclusive mode groups. The classes that currently declare `xor_group = "pads"` are:

- `MelodicMode`
- `RhythmicMode` (inherits from `MelodicMode`)
- `SliceNotesMode` (inherits from `MelodicMode`)
- `MackieControlMode`
- `TrackTriggeringMode` (defined but **not instantiated** by `app.py` â€” see notes below)

### Mode inheritance

Several modes are subclasses of other modes, not direct subclasses of `LogicMode`. Editing a parent silently changes the children:

- `RhythmicMode(MelodicMode)` â€” drum pad layout reuses melodic plumbing.
- `SliceNotesMode(MelodicMode)` â€” slice grid reuses melodic plumbing.
- `ScaleMenuMode(SettingsMode)` â€” scale picker reuses the settings-page UI patterns.
- `LogicInterface(LogicMode)` â€” yes, the Logic facade itself is a `LogicMode` (though it is not added to `active_modes` like a normal mode).

### Key Modes

- `main_controls_mode.py`: maps Push transport/control buttons to Logic actions through MCU or keybindings.
- `melodic_mode.py`: melodic pad layout, scales, octave shifts, fixed velocity, aftertouch, and scale collapse.
- `rhythmic_mode.py`: drum/rhythm pad layout, subclass of `MelodicMode`.
- `slice_notes_mode.py`: sequential note grid for slice-style triggering, subclass of `MelodicMode`.
- `track_selection_mode.py`: builds track metadata from `track_listing.json` and instrument definition JSON files.
- `midi_cc_mode.py`: maps encoders/display to per-instrument MIDI CC controls.
- `preset_selection_mode.py`: sends program changes and manages favorites.
- `settings_mode.py`: Push-side configuration pages and persistence.
- `scalemenu_mode.py`: scale selection UI, subclass of `SettingsMode`.
- `repeat_mode.py`: repeat-related Push mode.
- `help_mode.py`: help overlay/pages, reads `button_docs.json`.
- `mackie_control_mode.py`: Push-side Mackie mixer/control surface mode.
- `track_triggering_mode.py`: clip-grid trigger mode. **Defined but never instantiated** by `app.py` in the current tree â€” treat as dormant/legacy. If you wire it up, note it claims `xor_group = "pads"` and will exclude melodic/rhythmic/slice/mackie modes.

### Logic and MCU Integration

- `logic_interface.py` is the high-level Logic command facade. It prefers MCU for supported actions and falls back to keybindings. (Technically `LogicInterface(LogicMode)`, see Mode inheritance above.)
- `logic_keystrokes.py` loads `push2_keymap.json` and uses `pynput` to send macOS keystrokes.
- `logic_mcu_manager.py` opens Mackie Control MIDI ports, tracks transport, faders, pan, meters, LCD/scribble-strip text, selected track, and button state. Large file (~52 KB).
- `mcu_mode_detector.py` infers current Mackie assignment/submode from MCU traffic.
- `mcu_state.py` provides shared MCU state/snapshots.
- `logic_midi_listener.py` is the non-MCU fallback listener for basic Logic play/record state.

Which listener runs is controlled by `settings.json:use_mcu`. The MCU port name comes from `settings.json:mcu_port_name` (default `IAC Driver LogicMCU_In`).

Be careful with MCU note numbers, aliases, flip state, and assignment modes. Logic and different controller profiles may emit overlapping note IDs; this code contains normalization logic for that. MCU/Mackie constants (model IDs, SysEx prefixes, modifier notes, flip note number) all live in `definitions.py`, not in the MCU files themselves.

For diagnosing MCU/display issues, the debug HUD toggles in `definitions.py` (`MC_DRAW_DEBUG`, `MC_DEBUG_GUIDES`, `MC_SCRIBBLE_TEST`, etc.) and the `debug_mcu` flag in `settings.json` are the main levers.

### Display/UI

Display rendering uses `cairocffi` and writes RGB565 frames to the Push 2 display.

- `display_utils.py`: shared text/notification/help drawing helpers.
- `splash_screen.py`: startup display.
- `ui/base.py`: renderer dataclasses/protocols.
- `ui/legacy_gui.py`: legacy Mackie display renderer.
- `ui/mackie_gui_v2.py`: alternative (newer) Mackie display renderer.
- `ui/ring_styles.py`: ring rendering style mapping.
- `pad_meter.py`: `PadMeter` paints Logic's 0â€“127 level meters onto the Push 2 8Ă—8 pad grid using a `BLACK â†’ GREEN_LIGHT â†’ GREEN â†’ YELLOW â†’ ORANGE â†’ RED` ramp; caches last colors per pad to suppress duplicate MIDI traffic.

The GUI profile is selected by `settings.json:gui_profile`, currently typically `legacy` or `mackie_v2`.

## Coding Guidelines

- Preserve the existing mode-dispatch pattern. New behavior should usually live in a mode or in `LogicInterface`, not directly in global Push callbacks unless it changes event semantics.
- Keep hardware side effects explicit. MIDI sends, button LED changes, pad updates, and display redraw flags should be easy to trace.
- Use `app.pads_need_update`, `app.buttons_need_update`, and `app.display_dirty` where appropriate instead of forcing repeated heavy updates.
- Avoid sending duplicate MIDI/LED updates in fast paths. Several modules already cache button/pad colors to reduce traffic.
- Do not casually change `xor_group` values; doing so changes which modes can coexist.
- Do not overwrite user configuration in JSON files unless the task explicitly requires it.
- Keep instrument and device definition JSON schemas compatible with the current readers in `track_selection_mode.py` and `midi_cc_mode.py`.
- Be conservative with timing constants for long press, double press, MCU polling, and delayed actions. These affect physical controller feel.
- Prefer small, targeted changes. This app is stateful and hardware-dependent, so broad refactors are hard to validate.

## Validation

There is no formal automated test suite in the repository. Reasonable validation depends on the change:

- Syntax check edited Python files:

```bash
python -m py_compile app.py path/to/changed_file.py
```

- For wider Python edits:

```bash
python -m compileall .
```

Exclude `.venv`, cache directories, and IDE files when scanning manually.

Manual validation for behavior changes usually means running `./run.sh` with Push 2 and Logic configured, then checking the affected mode, MIDI routing, button LEDs, pads, display, and Logic response.

## Repository Hygiene

Ignore these when understanding or editing the project (safe to exclude from greps and tree views):

- `.venv/`
- `__pycache__/` (root and `ui/__pycache__/`)
- `.idea/`
- `.run/`
- `.DS_Store`
- `docs/` images (`*.jpg`, `*.png`) â€” documentation assets, not code
- generated `*.pyc`

`TestScripts/` contains standalone MCU/HUI probing and meter scripts (`Mackie.py`, `hui_test.py`, `mackie_meter_test.py`, `mcu_probe.py`) plus empty `mcu_log.csv`/`mcu_log.txt` capture files. They import `mido` / `rtmidi` directly and do **not** depend on `app.py`. Useful for hardware investigation; not part of the normal app startup.
